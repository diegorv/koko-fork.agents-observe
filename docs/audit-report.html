P0 · Fix now

01 P0

Unhandled rejection on startup repair

app/server/src/index.ts:15-30
Impact: High Complexity: Low Area: Server / startup
Problem
store.repairOrphans().then(...) at module scope has no .catch(). If repair throws (e.g. DB corruption, FS error) Node logs an unhandledRejection warning and the server proceeds with potentially unsafe state. No structured error surface.
Evidence
store.repairOrphans().then((result) => {
  const total = result.sessionsReassigned + ...
  if (total > 0) {
    console.log(`[startup] Repaired orphaned rows: ...`)
  }
})
Fix
Add .catch(err => { console.error('[startup] repair failed', err); process.exit(1) }) — or await the call inside an async bootstrap that the rest of the server depends on.
02 P0

Backup can silently overwrite the live database

app/server/src/routes/admin.ts:106-119
Impact: Critical (data loss) Complexity: Low Area: Server / admin
Problem
dbPath.replace(/\.db$/, ...) returns the input unchanged when config.dbPath does not end in .db (e.g. observe, or a custom path without an extension). The subsequent copyFileSync(dbPath, backupPath) then copies the file onto itself — or worse, fails partway and corrupts it — before clearAllData() runs.
Evidence
const backupPath = dbPath.replace(/\.db$/, `-${timestamp}.bak.db`)
try {
  copyFileSync(dbPath, backupPath)
} catch (err) {
  return apiError(c, 500, 'Failed to create database backup before reset', ...)
}
Fix
Build the backup path with path.join(path.dirname(dbPath), `${path.basename(dbPath, path.extname(dbPath))}-${timestamp}.bak.db`), or assert dbPath.endsWith('.db') and 400 otherwise. Also assert backupPath !== dbPath.
03 P0

Silent catch {} on WebSocket JSON parses

app/server/src/websocket.ts:21-40, app/client/src/hooks/use-websocket.ts:232-237
Impact: High Complexity: Low Area: WS transport
Problem
Both ends of the WS pipe wrap message parsing in try { JSON.parse(...) } catch {}. Malformed frames are dropped with no log, no metric, no client/server feedback. Whole categories of regressions (encoding bugs, partial frames, schema drift) become invisible.
Evidence
// server
ws.on('message', (raw) => {
  try {
    const msg: WSClientMessage = JSON.parse(raw.toString())
    ...
  } catch {}
})

// client
ws.onmessage = (wsEvent) => {
  try {
    const msg: WSMessage = JSON.parse(wsEvent.data)
    handleMessage(msg)
  } catch {}
}
Fix
Log the parse error (truncated payload + length). Optionally close the connection after N consecutive parse failures on either side.
04 P0

Unvalidated parseInt + unbounded limit

app/server/src/routes/sessions.ts:126-134
Impact: High (correctness + DoS) Complexity: Low Area: Server / events API
Problem
parseInt(c.req.query('limit')!), parseInt(c.req.query('offset')!) and parseInt(sinceParam) are sent straight to the DB layer with no isNaN check and no upper bound on limit. ?limit=999999 loads the entire event table into memory; ?since=abc yields NaN bound to a SQL parameter.
Evidence
const rows = sinceParam
  ? await store.getEventsSince(sessionId, parseInt(sinceParam))
  : await store.getEventsForSession(sessionId, {
      ...
      limit: c.req.query('limit') ? parseInt(c.req.query('limit')!) : undefined,
      offset: c.req.query('offset') ? parseInt(c.req.query('offset')!) : undefined,
    })
Fix
Centralise an integer-query helper: parsePositiveInt(value, { max?, default? }) that returns a typed result or 400s. Cap limit to e.g. 500 and reject NaN/negative values.
05 P0

JSON.parse in event-mapping loop has no try/catch

app/server/src/routes/sessions.ts:145-158
Impact: High (whole endpoint crash) Complexity: Low Area: Server / events API
Problem
JSON.parse(r.payload) and JSON.parse(r._meta) are called for every event row. A single corrupted row (truncated write, partial flush) throws synchronously and rejects the entire request. The response is all-or-nothing.
Evidence
const events: EventRow[] = rows.map((r) => {
  const base: EventRow = {
    ...
    payload: JSON.parse(r.payload),
  }
  ...
  if (requested.has('_meta')) base._meta = r._meta ? JSON.parse(r._meta) : null
  return base
})
Fix
Wrap each parse in a helper safeParse(json: string | null): unknown | null that returns null on failure and logs once. Surface the corrupt row's id in the log so a follow-up cleanup is possible.
06 P0

Dependabot config is a stub — no updates monitored

.github/dependabot.yml:7-11

07 P0

Hook wrapper always exits 0 — failures invisible

hooks/scripts/hook.sh:1-9
Impact: Medium-High Complexity: Low Area: Hooks / observability
Problem
The wrapper backgrounds the node CLI and unconditionally exit 0. If node is missing, the script path is wrong, or observe_cli.mjs throws on startup, Claude Code still sees a successful hook and the user loses observability silently — the very thing this product is supposed to provide.
Evidence
input=$(cat)
echo "$input" | node "$SCRIPT_DIR/observe_cli.mjs" hook > /dev/null 2>&1 &
exit 0
Fix
Validate command -v node and the existence of observe_cli.mjs before backgrounding. Write a tiny breadcrumb to ~/.claude/logs/observe-hook.log on failure. Keep the fast-exit behaviour for the success path.

P1 · Quick wins

08 P1

start.mjs uses npm install — lockfile drift

start.mjs:36-37
Impact: Medium Complexity: Low Area: Tooling
Problem
The local launcher runs npm install on every start, which can mutate package-lock.json. CI and the Dockerfile use npm ci. Result: local installs gradually drift from the reproducible build.
Evidence
// 1. Install dependencies
run('npm', ['install'], serverDir)
run('npm', ['install'], clientDir)
Fix
Switch to npm ci --no-audit --no-fund. Keep a just install recipe for the rare case where a dev wants to add a dep.
09 P1

Hot-path http helper swallows JSON errors

hooks/scripts/lib/http.mjs:34-38
Impact: Medium Complexity: Low Area: Hooks transport
Problem
On parse failure the helper returns the raw string as body. Callers expect an object — downstream type-confusion manifests as obscure crashes far from the source.
Fix
Return a clearly-typed shape: { status, body: null, parseError: e.message }. Log once with truncated raw text.
10 P1

Missing noUncheckedIndexedAccess

app/server/tsconfig.json, app/client/tsconfig.app.json
Impact: Medium Complexity: Low — Medium (depending on fallout) Area: Types
Problem
Both configs enable strict but omit noUncheckedIndexedAccess. Indexing arr[i] / map[k] returns T rather than T | undefined, hiding a broad family of off-by-one and missing-key bugs that show up at runtime.
Fix
Flip the flag, run tsc --noEmit, fix the fallout behind a small refactor PR. Most call sites will need an explicit guard or a non-null assertion with a comment.
11 P1

as any hides SessionStart payload shape

app/client/src/hooks/use-permission-mode-backfill.ts:42, 55
Impact: Medium Complexity: Low Area: Client types
Problem
Two e.payload as any casts to read permission_mode / permissionMode. If the SessionStart payload schema evolves on the server, the client compiles fine and breaks at runtime.
Fix
Declare type SessionStartPayload = { permission_mode?: string; permissionMode?: string } and cast through that. Better: add a tiny zod/typia parser at the WS boundary.
12 P1

Filter-config JSON.parse swallowed silently

app/server/src/storage/sqlite-adapter.ts:1360-1373
Impact: Medium Complexity: Low Area: Server / storage
Problem
A corrupt config column becomes {} with no log. Users see filters mysteriously stop working with no breadcrumb back to the bad row.
Fix
Log console.warn('[storage] filter ${id} has invalid config', err) in the catch arm. Consider quarantining the row with a flag instead of returning empty config.
13 P1

Consumer sweep timer not cleared on graceful shutdown

app/server/src/consumer-tracker.ts:12-36
Impact: Low-Medium Complexity: Low Area: Lifecycle
Problem
sweepTimer is only cleared inside the auto-shutdown branch of setTimeout. There is no SIGTERM/SIGINT handler that stops the interval, so the process can hold itself open one extra sweep tick under graceful shutdown and orphan a child of the test runner.
Fix
Export stopConsumerSweep() and call it from a top-level signal handler in app/server/src/index.ts alongside server.close().
14 P1

API client throws raw SyntaxError on non-JSON 5xx

app/client/src/lib/api-client.ts:66
Impact: Medium Complexity: Low Area: Client transport
Problem
return res.json() with no try/catch. A reverse proxy 502 (HTML body) or empty 504 body crashes the calling hook with a SyntaxError instead of the ApiError the rest of the UI handles.
Fix
Wrap parse in try/catch and re-throw as new ApiError(res.status, path, 'Invalid JSON response', raw.slice(0, 256)).

P2 · Architecture & performance

15 P2

extractResultText(toolResponse: any) — untyped tool-response handling

app/client/src/agents/claude-code/process-event.ts:392
Impact: Medium Complexity: Medium Area: Client domain
Problem
A central rendering helper accepts any and probes shape with optional chaining. As tool responses gain new shapes nothing surfaces statically, and the function quietly returns null on the ones it does not recognise.
Fix
Define a discriminated union of the response shapes Claude Code actually emits (string, { stdout }, { content: Array<{ type: string; text?: string }> }, etc.) and switch on the discriminant. Surfaces new shapes as exhaustiveness errors at compile time.
16 P2

Rewind-mode scroll listener closes over stale filteredEvents

app/client/src/components/event-stream/event-stream.tsx:248-274
Impact: Medium Complexity: Medium Area: Client / virtualizer
Problem
syncTimelineFromScroll closes over filteredEvents. The useEffect below re-attaches the scroll listener on every change, but in rewind mode — with rapid timeline jumps — the ‘old’ listener can still fire against the previous array between attach/detach.
Fix
Track latest events via a ref: const filteredEventsRef = useRef(filteredEvents), update in a sync effect, and have syncTimelineFromScroll read from the ref. Then the listener can be attached once for the lifetime of rewind mode.
17 P2

expandAllCounter effect drops two dependencies without comment

app/client/src/components/event-stream/event-stream.tsx:241-245
Impact: Medium Complexity: Low Area: Client / hooks
Problem
The effect calls expandAllEvents(filteredEvents.map(...)) but the dep array is just [expandAllCounter], no // eslint-disable rationale attached. The zustand action is stable today — but the filteredEvents snapshot the call uses is whichever render last committed, which may already be stale after a fast filter change.
Fix
Either (a) include filteredEvents in the deps and accept the extra fire, or (b) make expandAllEvents read the latest list from the store itself, removing the closure over component state.
18 P2

Schema migrations driven by PRAGMA table_info

app/server/src/storage/sqlite-adapter.ts (migration block ~44-68)
Impact: Medium Complexity: Medium Area: Storage / lifecycle
Problem
Schema upgrades are inferred from column presence rather than tracked in a migrations table. Idempotency is implicit. As soon as a second mutation depends on the first, the ordering becomes brittle and migration code starts asserting on database shape instead of declared version.
Fix
Add a schema_migrations(id INTEGER PRIMARY KEY, name TEXT, applied_at INTEGER) table. Name files 001_drop_columns.sql, 002_..., and apply in order, skipping rows already present.
19 P2

Sidebar drag leaks listeners on unmount mid-drag

app/client/src/components/sidebar/sidebar.tsx:31-58
Impact: Low-Medium Complexity: Low Area: Client / UI
Problem
handleMouseDown attaches mousemove and mouseup to document and removes them only inside the mouseup handler. If the component unmounts (or the tree re-keys) mid-drag, those listeners stay attached and continue to call into a stale setter closure.
Fix
Add a useEffect that returns a cleanup removing both listeners, or refactor to keep the ‘dragging’ flag in component state and bind/unbind via effect.

20 P3

Vite build output not content-hashed

app/client/vite.config.ts
Impact: Low-Medium Complexity: Low Area: Client / build
Problem
Default Vite output without an explicit hash strategy can leave stale clients talking to a newer server API across redeploys. The CHANGELOG calls out a v0.9.7 release path that touches Docker base images — cache busting matters there.
Fix
Set build.rollupOptions.output to [name].[hash].js for entries and chunks and [name].[hash][extname] for assets. Ensure index.html references the hashed bundle (Vite does this by default once hashes are enabled).
21 P3

GitHub Actions pinned to major only

.github/workflows/*.yml
Impact: Low Complexity: Low Area: CI / supply chain
Problem
actions/checkout@v5 and friends auto-pull any upstream v5.x.y. A compromised or buggy minor release flows in unannounced. Low likelihood, high blast radius if it ever bites.
Fix
Pin to full commit SHAs (preferred by SLSA & the GitHub security guide). Configure Dependabot github-actions updates so the SHAs stay current.
22 P3

Docker base image not pinned to a digest

Dockerfile:18-20
Impact: Low Complexity: Low Area: Infra
Problem
FROM node:24.10-slim follows the upstream tag and gets silently retagged. Comments in the file already note that pinning is preferable.
Fix
Resolve once with docker pull node:24.10-slim && docker inspect --format='{{.RepoDigests}}' node:24.10-slim and use FROM node:24.10-slim@sha256:.... Document the refresh cadence in the release runbook.
23 P3

No coverage thresholds in vitest configs

vitest.config.ts, app/server/vitest.config.ts, app/client/vitest.config.ts
Impact: Low Complexity: Low Area: Testing
Problem
No coverage block. Coverage can quietly shrink over time with no signal at PR review.
Fix
Run a baseline once (vitest run --coverage), then add coverage: { provider: 'v8', reporter: ['text', 'json-summary'], thresholds: { lines: 70, functions: 70, branches: 65, statements: 70 } } — numbers tuned to the baseline you measure.
